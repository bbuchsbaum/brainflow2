# System Sharpening Plan

_Goal_: Make cross‑layer features (like “sample value under cursor” and “load atlas → layer”) **boringly reliable** and easy to extend, by tightening ownership, unifying transports, and adding focused tests. This is meant to be actionable enough that future‑me can resume after a week without re‑deriving context.

---

## 0. Scope & Constraints

- Targeted to the **api‑bridge ↔ TS API ↔ UI** seam, not a rewrite of render_loop or loaders.
- Focus on:
  - Backend command permissions & discoverability.
  - Transport / invocation path (Tauri).
  - Atlas → layer lifecycle.
  - Sampling under cursor.
- Keep changes incremental and backwards‑compatible where possible; avoid breaking public APIs.

---

## 1. Unify Backend Access for TypeScript

### 1.1. Canonicalize the transport

**Problem**: We currently have two ways to hit Tauri:
- `ApiService` + `TauriTransport` using `@tauri-apps/api/core`.
- `@brainflow/api` helpers with their own `getInvoker()` logic, plus a dynamic fallback.

**Plan**:

1. **Design** a single abstraction that all UI code uses:
   - `CoreInvoker` interface (`(cmd: string, args?: Record<string, unknown>) => Promise<any>`).
   - A single implementation in `ui2/src/services/transport.ts` that wraps `TauriTransport.invoke`.
   - Re-export a helper from `transport.ts` so other modules (including bootstrapping) don’t import `@tauri-apps/api/core` directly.

2. **Wire @brainflow/api to CoreInvoker**:
   - Replace the direct `configureInvoker((cmd, args) => tauriInvoke(cmd, args))` call in `ui2/src/main.tsx` with a small helper `configureBrainflowApiInvoker()` in `ui2/src/services/transport.ts` that:
     - Obtains the shared `CoreInvoker` from `transport.ts`.
     - Passes it to `@brainflow/api`’s `configureInvoker`.
   - Keep `configureInvoker` as the public entry point in `@brainflow/api`, but **treat `ui2` bootstrapping as the canonical configuration site**.

3. **Deprecate internal fallback in @brainflow/api**:
   - In `packages/api/src/helpers.ts`, gate the dynamic `require('@tauri-apps/api/tauri')` / `window.__TAURI__` fallback behind a dev‑only warning:
     - If `getInvoker()` falls back, log once: “@brainflow/api is using legacy invoker fallback; call configureInvoker() at startup”.
   - Medium‑term: plan to remove the fallback and require configuration (not in this sprint).

4. **Gradually migrate direct `invoke` callers**:
   - For new UI code, prefer `CoreInvoker`/`getTransport()` over direct `@tauri-apps/api/core` usage (e.g. AtlasService can eventually be migrated).

**Outcome**: Any TS code (UI or packages) goes through the same path and uses the same Tauri `invoke`. No more “No Tauri invoke function configured” surprises, and fewer ad‑hoc imports of `@tauri-apps/api/core`.

## 2. Make Permissions First‑Class & Discoverable

### 2.1. Command inventory & permission mapping

**Problem**: New commands like `sample_layer_value_at_world` can be added to the plugin but silently blocked by ACL if we forget the permission entries.

**Plan**:

1. **Inventory commands**:
   - Script (Rust or small TS) to scan `core/api_bridge/src/lib.rs` for `#[tauri::command]` functions.
   - Output a list of command names and compare against:
     - `core/api_bridge/permissions/autogenerated/commands/*.toml` filenames.
     - `permissions.default.permissions` list in `core/api_bridge/permissions/default.toml`.

2. **Add missing permission files**:
   - We already added `sample_layer_value_at_world.toml`; run the inventory tool and:
     - Ensure every `#[command]` used by the UI has a `commands/*.toml` entry.
     - If a command is backend‑only or test‑only, annotate in the script to ignore it.

3. **Validate default permissions**:
   - Extend the script to assert that all UI‑used commands’ `allow-*` identifiers appear in `[default].permissions`.
   - Wire this as a **non‑blocking** CI check initially (warn, not fail).

4. **Document the rule**:
   - In `core/api_bridge/README.md` or `memory-bank/System_sharpening.md`:
     - “When you add a new `#[tauri::command]` that the UI invokes:
        1) add a `commands/<cmd>.toml`, 2) add `allow-*` to `default.toml`.”

**Outcome**: Adding a new backend command that the UI consumes is a **checklisted** operation; we won’t get “api-bridge.<cmd> not allowed” at runtime by accident.

## 3. Sharpen Sampling Under Cursor

### 3.1. Define a SamplingService boundary

**Problem**: Sampling is currently wired directly from `SliceViewCanvas` into `@brainflow/api` and then down into the bridge. Errors are surfaced ad‑hoc, and there’s no single owner for “sample value under cursor”.

**Plan**:

1. **Introduce a thin `SamplingService` in ui2**:
   - New file: `ui2/src/services/SamplingService.ts`.
   - Expose:
     ```ts
     export interface SampleOptions {
       layerId: string;
       world: [number, number, number];
     }

     export interface SampleResult {
       value: number | null;
       error?: Error;
     }

     export async function sampleLayerAtWorld(opts: SampleOptions): Promise<SampleResult>;
     ```

2. **Implementation**:
   - Internally call `sampleLayerValueAtWorld(layerId, world)` from `@brainflow/api`.
   - Centralize error logging:
     - If it fails, log a single structured error with:
       - layerId, world coords, raw error, and possibly a “hint” (mapping/permission issues).
     - Return `{ value: null, error }` instead of throwing.

3. **Update `SliceViewCanvas` to depend on SamplingService**:
   - Replace direct calls to `sampleLayerValueAtWorld` with `SamplingService.sampleLayerAtWorld`.
   - Treat `SampleResult.error` as “show `--` but don’t spam logs”; SamplingService owns the logging.

4. **Backend hardening**:
   - In `sample_layer_value_at_world` (Rust), keep current behavior but:
     - Add targeted `info!` logging when `layer_to_volume_map` doesn’t contain the layer id:
       - Log all keys and the incoming id once per session (throttle).

**Outcome**: All sampling logic and error reporting flows through one place; `SliceViewCanvas` just uses a simple API and updates the UI. Future features (e.g., sampling time series, label names) can extend SamplingService without touching view components.

### 3.2. Tests for sampling

**Unit/integration tests** (Rust side):

1. **Sample from synthetic 3D volume**:
   - In `core/api_bridge/tests/affine_transform_test.rs` or a new test module:
     - Create a small `DenseVolume3<f32>` with known values (e.g., value = x + 10*y + 100*z).
     - Register it in a `VolumeRegistry` with a fixed handle id.
     - Call `sample_world_coordinate(handle_id, world)` for coordinates that map to specific voxels.
     - Assert sampled values match expectations and that out‑of‑bounds returns 0.

2. **Sample via `sample_layer_value_at_world` path**:
   - Construct a BridgeState with:
     - `volume_registry` containing the synthetic volume.
     - `layer_to_volume_map` mapping a known `layer_id` to the handle id.
   - Call `sample_layer_value_at_world(layer_id, world, state)` and assert:
     - Success for valid coords.
     - `BridgeError::Input` when `world_coords.len() != 3`.
     - `BridgeError::Input` “No volume handle mapped…” when using an unknown layer id.

**Vitest tests** (TS side, limited to plumbing):

3. **SamplingService integration with mocked invoker**:
   - In `ui2/src/services/__tests__/SamplingService.test.ts`:
     - Use `configureInvoker` with a mocked invoke function that returns a known numeric value.
     - Assert `sampleLayerAtWorld` returns `{ value: 42 }` and that errors are handled as `{ value: null, error }`.

## 4. Consolidate Atlas → Layer Ownership

### 4.1. Strengthen `AtlasService.loadAtlasAndCreateLayer` as the canonical path

**Problem**: Atlas layers can be created from multiple UI pathways (menu, panel), all of which can call into VolumeLoadingService. Without strong ownership, it’s easy to end up with duplicate layers for the same atlas config.

**Plan**:

1. **Affirm `loadAtlasAndCreateLayer` as the only supported path**:
   - Audit usages:
     - `useAtlasMenuListener` ✓
     - `AtlasConfigModal` ✓
     - Ensure no one else calls `AtlasService.loadAtlas` + `VolumeLoadingService.loadVolume` directly for atlases.

2. **Document invariants** in `ui2/src/services/AtlasService.ts`:
   - “All atlas loads that should create layers must go through `loadAtlasAndCreateLayer`.”

3. **Preserve and comment the dedupe guard in `VolumeLoadingService`**:
   - Keep the atlas‑aware dedupe:
     - Reuse existing layer when `(atlasMetadata.id, space, resolution, n_regions)` match.
   - Add a short comment:
     - “StrictMode / double‑mount safety: even if multiple callers request the same atlas, we only create one layer.”

4. **Optional UI refinement**:
   - In `AtlasPanel`, when a preset is selected from the menu:
     - Consider a UI state where “Load” is disabled if a layer already exists for the currently selected atlas config (read from the layer store’s `atlasMetadata`).

### 4.2. Tests for atlas deduplication

**TS side (Vitest)**:

1. **Atlas dedupe test**:
   - Mock:
     - `AtlasService.loadAtlas` to return a fixed `volume_handle_info` and `atlas_metadata`.
     - `VolumeLoadingService.getVolumeLoadingService()` to use the real implementation but with a mocked transport that doesn’t actually hit Tauri.
   - Simulate two calls to `AtlasService.loadAtlasAndCreateLayer` with the same `AtlasConfig`.
   - Assert:
     - `VolumeLoadingService.loadVolume` is called twice.
     - But after both calls, `useLayerStore.getState().layers` contains **one** layer with matching `atlasMetadata`.
     - The second call returned the same `Layer` object (id and metadata equal).

If mocking the full path is too heavy, test just `VolumeLoadingService.loadVolume` with a synthetic config where `source: 'atlas'` and `atlasMetadata` are fixed, and call it twice.

## 5. Improve Observability & Error Surfacing

### 5.1. Standardize logging for cross‑layer commands

**Plan**:

1. **Sampling errors**:
   - Already addressed via `SamplingService`. Ensure we:
     - Log once per distinct `(command, layerId, coord)` triple (use a small in‑memory cache to throttle).

2. **Atlas load errors**:
   - In `AtlasService.loadAtlas`, we already translate `BridgeError` into readable messages.
   - Add a small helper to attach the requested `(parcels, networks, resolution)` to error logs so we can see when the requested combo isn’t actually available on disk.

3. **Status bar errors (optional)**:
   - Consider a low‑priority “Status” slot where critical backend issues (like repeated sampling failures) are surfaced briefly (e.g., “Sampling disabled: backend error”).
